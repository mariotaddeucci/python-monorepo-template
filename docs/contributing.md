# Contributing

Thank you for your interest in contributing to this project. This guide covers everything you need to get started: environment setup, development workflow, code guidelines, and how the monorepo is structured.

---

## Requirements

- [uv](https://docs.astral.sh/uv/) — the only supported package manager
- Git
- Python 3.11+ (managed automatically by uv via `.python-version` files)

---

## Project Setup

### 1. Clone the repository

```bash
git clone https://github.com/mariotaddeucci/python-monorepo-template.git
cd python-monorepo-template
```

### 2. Install all dependencies

```bash
uv sync
```

This installs the full workspace — all packages and all dev dependencies — into a single shared `.venv` at the root.

### 3. Install pre-commit hooks

```bash
uv run pre-commit install
```

Hooks run automatically on every `git commit`. To run them manually over all files:

```bash
uv run pre-commit run --all-files
```

## Repository Structure

```
python-monorepo-template/
├── pyproject.toml                          # Workspace root: ruff, pytest, taskipy, dev deps
├── uv.lock                                 # Lockfile (commit this)
├── mkdocs.base.yml                         # Shared MkDocs config (inherited by packages)
├── mkdocs.yml                              # Root documentation portal
├── docs/                                   # Root portal content
├── scripts/
│   ├── run_task.py                         # Task runner with auto-discovery and progress bar
│   └── build_docs.py                       # Documentation build/serve orchestrator
└── packages/
    ├── my-sample-package/                  # Proxy package — re-exports all providers
    ├── my-sample-package-provider-core/    # Python >=3.11, module: core
    ├── my-sample-package-provider-app/     # Python >=3.12, module: app
    └── my-sample-package-provider-standalone/ # Python >=3.13, module: standalone
```

Each package follows this internal layout:

```
packages/<name>/
├── pyproject.toml       # Package metadata and tasks
├── .python-version      # Minimum Python version for the package
├── README.md
├── src/<module>/        # Source code
│   └── _version.py      # Auto-generated by hatch-vcs — never edit
├── tests/               # Test files
├── mkdocs.yml           # Package-level docs config (inherits mkdocs.base.yml)
└── docs/                # Package documentation content
```

## Development Workflow

### Running tasks across all packages

```bash
uv run task test       # Run all tests
uv run task lint       # Lint all packages
uv run task autofix    # Auto-fix lint issues
uv run task format     # Format code
```

The task runner (`scripts/run_task.py`) auto-discovers which packages define each task, shows a `tqdm` progress bar, and prints a PASS/FAIL summary with error details on failures.

### Running tasks for a single package

```bash
uv run --directory packages/my-sample-package-provider-core task test
uv run --directory packages/my-sample-package-provider-app task lint
```

> **Never** use `uv run --package <name> task <task>` — this causes infinite recursion. Always use `--directory`.

### Building and serving documentation

```bash
uv run task docs          # Build all package docs, then serve the root portal
uv run task docs-build    # Build all docs without serving
```

Each package's site is output to `site/packages/<name>/`. The root portal at `site/` links to all package subsites.

## Code Style

All style rules are enforced by [Ruff](https://docs.astral.sh/ruff/) and the pre-commit hooks.

| Rule | Value |
|------|-------|
| Line length | 120 characters |
| Indentation | 4 spaces |
| String quotes | Double (`"`) |
| Type annotations | Required on all public functions and methods |
| Generics | Use built-ins: `list[str]`, `dict[str, int]` (not `List`, `Dict`) |
| Docstrings | Google style on all public API |
| `__init__` return | Must declare `-> None` |

### Naming conventions

| Element | Convention | Example |
|---------|-----------|---------|
| Modules / functions | `snake_case` | `process_data` |
| Classes | `PascalCase` | `DataModel` |
| Constants | `UPPER_SNAKE` | `MAX_RETRIES` |
| Private members | `_leading_underscore` | `_internal_helper` |

### Error handling

- Always raise specific exceptions (`ValueError`, `RuntimeError`, etc.) with descriptive messages.
- Never use bare `except:`.
- Never use `assert False` — use `pytest.raises` in tests.
- Guard optional dependencies with `try/except ImportError`.

---

## Testing Guidelines

- Use plain functions (`def test_*`), never test classes.
- Use `pytest.raises` for expected exceptions — never `assert` on exception strings directly.
- Name tests to describe the behaviour being verified: `test_clamp_returns_max_when_above_range`.

```bash
# Run tests with coverage
uv run --directory packages/my-sample-package-provider-core task test
```

## Pre-commit Hooks

The following hooks run on every commit:

| Hook | Purpose |
|------|---------|
| `ruff-check --fix` | Lint and auto-fix Python code |
| `ruff-format` | Format Python code |
| `check-json` | Validate JSON files |
| `check-toml` | Validate TOML files |
| `check-yaml` | Validate YAML files |
| `end-of-file-fixer` | Ensure files end with a newline |
| `trailing-whitespace` | Remove trailing whitespace |
| `mixed-line-ending` | Normalize line endings |
| `pyproject-fmt` | Normalize `pyproject.toml` formatting |
| `semgrep` | Static analysis security and correctness checks |

> `pyproject-fmt` may reorder keys and normalize indentation in `pyproject.toml` automatically. This is expected — just `git add` the reformatted file and commit again.

## Versioning

Versions are managed dynamically via [hatch-vcs](https://github.com/ofek/hatch-vcs). Do **not** hard-code version numbers anywhere.

Tag format: `<package-name>:<version>` — e.g. `my-sample-package-provider-core:1.2.3`.

The file `src/<module>/_version.py` is auto-generated at build time. Never edit or commit it.

---

## Adding a New Package

1. Create `packages/<new-package-name>/` with:
    - `pyproject.toml` (with `hatch-vcs` dynamic versioning and taskipy tasks: `test`, `lint`, `autofix`, `format`, `docs`)
    - `src/<module>/` with `__init__.py` and `_version.py` excluded from version control
    - `tests/` with at least one test file
    - `.python-version` with the minimum Python version
    - `README.md`
2. Add the MkDocs setup:
    - `mkdocs.yml` inheriting `../../mkdocs.base.yml`
    - `docs/index.md` and `docs/api/reference.md`
3. The root task runner and docs builder auto-discover packages — no changes needed to root `pyproject.toml` taskipy tasks.

---

## Common Pitfalls

- **Never edit `_version.py`** — it is auto-generated by `hatch-vcs` and must not be committed.
- **Dev dependencies** go in `[dependency-groups] dev` in the **root** `pyproject.toml`, not in individual package `[project.optional-dependencies]`.
- **Always use `uv`** — never `pip`, `python -m pip`, or `poetry`.
- **Use `--directory`** for single-package tasks, never `--package`.
